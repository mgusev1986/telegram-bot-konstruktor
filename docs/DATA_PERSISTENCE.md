# Сохранность данных при деплое

## Что сделано для сохранности

1. **Backfill при старте** — при каждом запуске бота все цепочки, рассылки и связанные данные с `botInstanceId = null` или привязанные к другому экземпляру с тем же `BOT_USERNAME` автоматически мигрируются к текущему боту. Это решает ситуацию, когда после деплоя «пропадали» цепочки из-за смены внутреннего ID бота.

2. **Бэкап перед деплоем** — скрипт `deploy-hetzner.sh` перед обновлением создаёт снимок БД через `server-backup-rotating.sh`. При проблемах можно восстановить из `/root/bot-backups/`.

3. **Тома Docker** — PostgreSQL хранит данные в томе `postgres_data`, который не удаляется при перезапуске контейнера бота.

## Данные сохраняются

Цепочки писем (drip-кампании), автосерии, рассылки, push-уведомления, меню, пользователи и платежи хранятся в **PostgreSQL** в Docker volume `postgres_data`.

При деплое (`npm run deploy`) выполняется только:
- пересборка контейнера **bot**
- перезапуск контейнера **bot**

Контейнеры **postgres** и **redis** при этом не пересоздаются. Их тома (`postgres_data`, `redis_data`) остаются на месте, данные сохраняются.

## Что именно сохраняется

| Данные | Где хранится |
|--------|--------------|
| Цепочки писем (drip-кампании) | PostgreSQL |
| Рассылки (broadcasts) | PostgreSQL |
| Push-уведомления, напоминания | PostgreSQL |
| Меню, разделы, кнопки | PostgreSQL |
| Пользователи, рефералы | PostgreSQL |
| Платежи, подписки | PostgreSQL |
| Очереди задач (BullMQ) | Redis (пересоздаются при рестарте) |

## ⚠️ Когда данные могут потеряться

1. **`docker compose down -v`** — флаг `-v` удаляет тома вместе со всеми данными. **Никогда не используйте `-v`** при обычной остановке.

2. **Удаление тома вручную:**
   ```bash
   docker volume rm telegram-bot-konstruktor_postgres_data  # опасно!
   ```

3. **Полная переустановка** — если вы удалили проект и подняли заново, данные теряются.

## Защита: бэкапы

Настроен скрипт `server-backup-rotating.sh` — каждые 2 часа создаётся бэкап БД, .env и schema в `/root/bot-backups/`. Рекомендуется дополнительно:

1. **Проверить cron:** `ssh root@СЕРВЕР "crontab -l"`
2. **Выгружать бэкапы вовне** (Hetzner Object Storage, другой сервер) — см. [BACKUP.md](./BACKUP.md)

## Напоминания (push) и привязка к боту

Правила напоминаний о бездействии (`InactivityReminderRule`) теперь привязаны к экземпляру бота (`botInstanceId`). Это гарантирует, что:

- Напоминания сохраняются в PostgreSQL и **не теряются** при деплое
- Каждый бот видит только свои напоминания
- Запланированные задачи (`ScheduledJob`) восстанавливаются при старте через `recoverPendingJobs`
- Redis/BullMQ пересоздаются при рестарте, но задачи берутся из PostgreSQL и снова добавляются в очередь

Если после деплоя напоминания «пропали», проверьте:

1. Что тома Docker (`postgres_data`, `redis_data`) не удалялись
2. Что миграция `20260322120000_inactivity_reminder_rules_bot_instance` применена (`npx prisma migrate deploy`)
3. Что вы заходите в того же бота (в multi-bot среде напоминания привязаны к конкретному боту)

## Если цепочки не отображаются

Если после деплоя в «Мои цепочки» пусто, возможные причины:

1. **Цепочки не были созданы** для этого бота — проверьте, что заходите в того же бота и с тем же аккаунтом.
2. **Несколько ботов** — если в бэкофисе несколько экземпляров бота, цепочки привязаны к конкретному боту.
3. **Права** — нужна роль с доступом к админке и автосериям.

Данные в БД при обычном деплое **не удаляются**.
